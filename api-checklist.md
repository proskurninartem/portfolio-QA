# Чек-лист: Тестирование REST API

**Объект:** Reqres.in REST API  
**Автор:** Артём Проскурнин  
**Версия:** 1.0

---

## 1. HTTP-методы и статус-коды

- [x] GET-запрос возвращает `200 OK` при наличии ресурса
- [x] GET-запрос возвращает `404 Not Found` при отсутствии ресурса
- [x] POST-запрос возвращает `201 Created` при успешном создании
- [x] PUT-запрос возвращает `200 OK` при успешном обновлении
- [x] PATCH-запрос возвращает `200 OK` при частичном обновлении
- [x] DELETE-запрос возвращает `204 No Content` при успешном удалении
- [x] Запросы с некорректными данными возвращают `400 Bad Request`
- [x] Запросы на несуществующий ресурс возвращают `404`
- [ ] Неавторизованные запросы возвращают `401 Unauthorized` *(не применимо для Reqres.in)*

---

## 2. Структура ответа (контрактное тестирование)

- [x] Тело ответа в формате JSON
- [x] Заголовок `Content-Type: application/json` присутствует
- [x] Обязательные поля присутствуют в ответе (`id`, `email`, `first_name`, `last_name`)
- [x] Типы данных полей соответствуют документации (string/number/array)
- [x] Поле `id` — целое положительное число
- [x] Поле `email` — валидный email-формат
- [x] Поле `avatar` — валидный URL
- [x] Поле `createdAt` / `updatedAt` — формат ISO 8601
- [ ] Отсутствуют лишние поля, не описанные в документации

---

## 3. Пагинация

- [x] Параметр `page` работает корректно
- [x] Параметр `per_page` отражает количество элементов в `data`
- [x] Поле `total_pages` рассчитано верно (`total / per_page`)
- [x] Страница вне диапазона возвращает пустой массив
- [x] При `page=1` и `page=2` возвращаются разные наборы данных
- [ ] Параметр `per_page` принимается как query-параметр и изменяет количество записей

---

## 4. Валидация входных данных

- [x] Пустое тело запроса обрабатывается (POST)
- [x] Отсутствие обязательных полей вызывает ошибку
- [x] Слишком длинные значения полей обрабатываются
- [x] Специальные символы в полях (`<script>`, `DROP TABLE`) не вызывают ошибок сервера
- [x] `null` в значении поля обрабатывается корректно
- [ ] Числовые поля не принимают строковые значения *(частичный FAIL — см. BUG-001)*

---

## 5. Заголовки запроса

- [x] `Content-Type: application/json` передаётся в POST/PUT/PATCH
- [x] Запрос без `Content-Type` обрабатывается
- [x] `Accept: application/json` работает корректно
- [ ] Неверный `Content-Type` вызывает `415 Unsupported Media Type` *(сервер принимает любой)*

---

## 6. Производительность (базовая)

- [x] Время ответа GET-запроса < 2000 мс
- [x] Время ответа POST-запроса < 3000 мс
- [ ] Время ответа при 10+ одновременных запросах стабильно *(не тестировалось)*
